## Рекомендуемый порядок прохождения

### 1. Классический пример с фигурами

```bash
python 01_classic_shapes.py
```

В этом файле есть два класса:

* `Rectangle`;
* `Circle`.

У каждого класса есть метод `area()`.

Функция `print_area()` не знает, какую именно фигуру получила. Она просто вызывает общий метод:

```python
shape.area()
```

Это базовая идея полиморфизма: разные объекты можно использовать одинаково, если у них есть общий способ взаимодействия.

### 2. Проблема с `if/elif`

```bash
python 02_bad_task_renderer.py
```

В этом файле задача из условного Task Manager отображается в разных форматах:

* `text`;
* `markdown`;
* `html`.

Проблема в том, что функция `render_task()` сама знает все форматы и выбирает поведение через `if/elif`.

Чтобы добавить новый формат, например `json`, нужно менять старую функцию и добавлять новую ветку.

Главная мысль: проблема не в самом `if`, а в том, что одна функция начинает отвечать за слишком много вариантов поведения.

### 3. Рефакторинг к полиморфизму

```bash
python 03_polymorphic_task_renderer.py
```

В этом файле разные форматы вынесены в отдельные классы:

* `TextRenderer`;
* `MarkdownRenderer`;
* `HtmlRenderer`.

У каждого класса есть общий метод:

```python
render(task)
```

Функция `render_task()` больше не знает про конкретные форматы:

```python
def render_task(task, renderer):
    return renderer.render(task)
```

Теперь новый формат можно добавить через новый класс, не переписывая центральную функцию.

### 4. Утиная типизация

```bash
python 04_duck_typing.py
```

В Python объекту не обязательно наследоваться от общего родителя, чтобы его можно было использовать в общем коде.

Если объект имеет нужный метод, код работает.

Если метода нет, ошибка появится во время выполнения:

```python
AttributeError
```

Это называют утиной типизацией: важен не формальный тип объекта, а то, умеет ли он делать нужное действие.

### 5. Когда `if` — это нормально

```bash
python 05_when_if_is_ok.py
```

Полиморфизм не означает, что нужно избавиться от всех условий.

`if` нормально использовать для простых проверок:

* список задач пустой;
* пользователь не авторизован;
* у пользователя нет прав;
* входные данные некорректны.

Проблема начинается, когда `if/elif` хранит постоянно растущий список вариантов поведения.

## Какие темы можно отработать

### Классический полиморфизм

```python
def print_area(shape):
    print(shape.area())
```

Один вызов `area()`, но разные классы считают площадь по-разному.

### Общий интерфейс

```python
def render_task(task, renderer):
    return renderer.render(task)
```

Функция работает с любым объектом, если у него есть метод `render()`.

### Утиная типизация

Python не требует общего родителя или конкретного типа. Он просто пытается вызвать нужный метод в момент выполнения.

### Рефакторинг `if/elif`

До:

```python
def render_task(task, format_name):
    if format_name == "text":
        ...
    elif format_name == "markdown":
        ...
    elif format_name == "html":
        ...
```

После:

```python
def render_task(task, renderer):
    return renderer.render(task)
```

### Расширение поведения

Новый формат добавляется новым классом:

```python
class JsonRenderer:
    def render(self, task):
        ...
```

Старая функция `render_task()` не меняется.

## Важное про JSON в примере

В практическом задании JSON-строка собирается вручную специально для простоты.

В реальном коде обычно используют модуль `json`:

```python
import json
```

Но в этом вебинаре цель не в работе с JSON, а в том, чтобы увидеть принцип: новое поведение можно добавить через новый объект с тем же интерфейсом.

## Главная идея вебинара

Полиморфизм помогает писать код, который работает не с конкретными классами, а с ожидаемым поведением.

Если объект умеет делать нужное действие, например `render(task)` или `match(task)`, его можно использовать в общем коде.

Хороший признак:

> новый вариант поведения добавляется новым классом.

Плохой признак:

> центральная функция постоянно растёт из-за новых веток `elif`.
